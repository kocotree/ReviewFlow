# ReviewFlow 稳定性与模块化改进计划

## 1. 计划目标

本计划将当前代码审查发现的问题转化为分阶段整改任务。实施顺序以状态正确性为先，其次是异步性能和模块解耦，最后完成资源安全、部署与文档治理。

完成后应满足：

- 首次提交自动评分，首次评分后的普通记录编辑不自动触发评分。
- 用户完成附件或在线文档修改后，通过消息卡片按钮显式触发重新评分。
- 同一次按钮操作只产生一个评分任务，重复点击不会并行评分。
- 旧评分结果不会覆盖评分过程中产生的新内容。
- 外部调用或写回失败后，记录不会永久停留在“评分中”。
- 在线文档无需监听内容变化，用户点击按钮时统一读取最新内容。
- 非法 AI 响应不会作为 0 分结果消耗修改轮次。
- 飞书同步 SDK 不再阻塞 FastAPI 事件循环。
- 核心流程可以使用 Fake Client 完成自动化测试。
- 编排器只负责状态流转和步骤协调。

## 2. 实施原则

1. 先增加回归测试，再修改状态机。
2. 每个阶段独立提交，并保持服务可运行。
3. 优先采用进程内的简单实现，确认需要多实例后再引入 Redis 或任务队列。
4. 外部系统错误必须保留类型信息，不能统一压缩成 `None` 或 `False`。
5. 业务规则与技术重试机制分离。

## 2.1 多维表字段硬约束

本次改造不得新增以下多维表字段：

- `评分开始时间`
- `评分触发来源`
- `文档缓存版本`

相关技术需求按以下方式实现，不扩张业务表结构：

- 评分开始时间：清道夫使用飞书记录自带的系统更新时间（`last_modified_time`）判断“评分中”停留时长；该时间由写入“评分中”自然更新，不另建业务字段。
- 评分触发来源：只作为 `request_score()` 的任务命令属性和结构化日志上下文存在，用于区分首次事件、用户重评和管理员重试，不写回多维表。
- 文档缓存版本：不保存版本字段。缓存只由本次总 PDF 的成功合并转写生成，并在每次完成评分时覆盖；采集或转写失败时保留旧值但绝不读取为评分输入。

## 2.2 已确认的评分触发模型

评分采用“首次自动、后续手动”的触发方式：

1. 用户首次提交需求，记录处于“待评分”，Bitable 事件自动触发评分。
2. 首次评分完成后，状态进入“已通过”“未通过”“已驳回”或“评分异常”。
3. 处于“未通过”时，用户可以任意增删附件、分多次上传文件或修改在线文档；这些普通编辑事件只更新内容，不触发评分。
4. 用户确认修改完成后，在机器人消息卡片中点击“重新评分”。
5. 按钮回调读取记录的最新内容并提交一次评分任务。
6. 附件格式不支持、内容为空等可修复场景也进入“未通过”，并发送带“重新评分”按钮的卡片。

推荐状态流：

`待评分 → 评分中 → 已通过 / 未通过 / 评分异常`

`未通过 → 用户修改内容 → 点击重新评分 → 评分中`

这套模型不需要检测在线文档是否发生变化，也不需要对用户的每次附件上传做防抖或自动合并。

## 4. 阶段二：实现显式重评触发与状态一致性

优先级：P0

### 4.0 先架网：发送侧熔断（动状态机之前完成）

在重写状态机之前，先补一个只在“代码自转”时才会触发的发送侧熔断，防止本次重构期间复发的回声循环把整个机器人打爆（历史上已发生两次）。这不是对用户的限流——真实用户永远不会在数分钟内让同一条记录发出几十张卡。

- [x] 同一记录在 M 分钟内被发卡超过 N 次（如 5 分钟 20 次）即停发并触发管理员告警。
- [x] 熔断只作用于发送侧，不给正常提报加任何门槛。
- [x] 该项作为整个改造的第一个提交，排在任何状态机改动之前（先架网再拆）。

### 4.1 收敛自动触发条件

- [x] Bitable 记录事件只在状态为“待评分”时申请自动评分。
- [x] “未通过”状态下的文本、附件和文档链接编辑不申请评分。
- [x] 服务写回“评分中”“未通过”或“已通过”产生的事件直接忽略。
- [ ] 空内容和格式不支持等可修复场景统一写为“未通过”，不恢复为“待评分”。
- [ ] 删除依赖内容指纹识别写回回声的复杂逻辑。

### 4.2 增加卡片重新评分按钮

- [ ] 在未通过卡片中同时提供“查看并修改”和“重新评分”按钮。
- [ ] 在附件格式错误、无可评审内容卡片中增加“重新评分”按钮。
- [ ] 如需人工恢复 AI 异常，在群告警卡片中增加管理员“重试评分”按钮。
- [x] 按钮使用飞书卡片交互回调，不使用带副作用的普通 GET 链接。
- [x] 回调 payload 只携带动作类型和记录定位信息。
- [x] 回调成功后向用户返回“已提交重新评分”的提示。
- [x] 评分已在进行时返回“正在评分，请勿重复点击”。

### 4.3 增加卡片动作回调入口

- [x] 增加独立的卡片动作回调路由，并复用飞书验签和解密逻辑。
- [x] 校验 `action=rescore`，拒绝未知动作。
- [x] 重新读取记录并校验点击者身份：
  - 用户重评：点击者必须是消息接收人（私聊 p2p，接收人唯一，天然成立）。
  - 管理员重试（群告警卡片）：以群成员身份授权，不做 open_id 名单校验（依赖内部小组可信，见 §12 风险接受）。
- [x] 只允许“未通过”等可恢复状态进入重新评分。
- [x] 使用飞书事件 ID或回调 ID实现点击幂等（防同一次点击被“至少投递一次”重投）。
- [x] 用户重评与管理员重试走同一个 `request_score()`、同一把 §4.4 记录锁、同一套幂等；管理员不得另开捷径。
- [x] 触发类型只随任务命令在进程内传递并写入日志，不新增 `评分触发来源` 字段。
- [ ] 卡片点击后置灰仅作为 UX，不承担并发去重职责（置灰在一次网络往返之后才生效，挡不住已在途的第二份回调，真正拦并发的是记录锁 + 幂等）。
- [x] 回调只负责申请任务，实际评分仍进入统一协调器。

### 4.4 每记录串行协调

- [x] 使用 `app_token + table_id + record_id` 作为完整记录键。
- [x] 初次事件和按钮回调都调用统一的 `request_score()`。
- [x] 同一记录已有任务时拒绝或合并重复请求，不并行执行。
- [ ] 写回评分前校验本任务是否仍是该记录的在岗任务（fencing，见 §4.6）；非在岗任务的结果直接作废退出，禁止僵尸任务覆盖新评分。
- [ ] 不再在写回前做“内容版本”校验：采用点击即等待模型，评分期间不存在并发编辑（详见 §4.6 走位 C）。

### 4.5 集中管理状态转换

- [ ] 封装统一的状态转换方法，并检查所有写回结果。
- [ ] 设置“评分中”失败时立即终止，不调用 AI。
- [ ] 进入“评分中”后的异常必须恢复旧状态，或进入明确的可重试异常状态。
- [ ] 最终写回失败时保留重试条件。
- [ ] 修改轮次存于多维表字段（与通知频控是两坨独立状态）；`+1` 只在流程走到终态（通过/未通过）时发生，中途任何技术异常不计数。
- [ ] 收尾顺序钉死为“先算轮次、再选状态与卡片”：先算出本次完成后的新轮次 → 若新轮次 == 上限，终态直接写“已驳回”，发“已达上限，请联系开发者”的无按钮卡片 → 否则才写“未通过”并发带“重新评分”按钮卡片。轮次运算必须在选状态 / 选卡片模板之前，才是“立即驳回”而非“下次点击才驳回”。
- [ ] “重新评分”按钮是否出现由当前轮次动态决定，达到上限的记录不再渲染该按钮。

### 4.6 删除内容指纹机制，改用 fencing 防僵尸写回（走位 C）

采用点击即等待模型：用户点“重新评分”后就是在等结果，不会一边等一边改同一份文档，因此“评分期间并发编辑”这个场景不存在，防它的守卫全部删除。但清道夫（§5）会“复位旧任务 + 重新入队”，必然制造“新旧两个任务并存”的窗口，僵尸任务写回是真实风险，需用 fencing 挡住——它防的是任务并发，不是用户编辑，不能跟着 C 一起删。

- [ ] 写回回声通过状态门控处理，不再依赖 `_scored_sig`。
- [ ] 彻底删除内容签名机制：内容签名在新模型下已无任何用途（不再用于检测评分期间内容变化）。
- [ ] 删除“评分期间内容变化则本次作废”的守卫（并发编辑在点击即等待模型下不存在）。
- [ ] 写回前改为校验 fencing（每次入队自增的整数 / 任务身份）：比对任务注册表中该记录当前登记的在岗任务，非在岗任务的结果一律作废退出，杜绝清道夫复位后旧任务诈尸覆盖新评分。
- [ ] 按钮点击后始终重新读取在线文档和全部附件。

## 5. 阶段三：管理 Webhook 后台任务生命周期

- [x] 任务完成后从注册表中自动移除。
- [x] 服务关闭时停止接收新任务，并在超时范围内等待现有任务结束。
- [x] 区分事件已接收、任务已开始和任务已完成三种状态。

第一步采用进程内任务注册表；只有在多实例或不能接受重启丢任务时再引入 Redis、Arq、RQ 或 Celery。

### 崩溃恢复清道夫（非优雅关闭场景）

`drain` 只覆盖优雅关闭；OOM/驱逐/SIGKILL/掉电不给 drain 机会，会把记录永久冻在持久层的“评分中”（内存锁、任务表、幂等缓存全部丢失，唯独 Bitable 的“评分中”还在）。新模型下编辑不自动触发、按钮回调拒绝“评分中”、写回事件被忽略——三条自愈路全堵死，必须有清道夫兜底。

- [ ] 周期性扫描 Bitable 中处于“评分中”的记录，并读取记录的系统 `last_modified_time` 元数据。
- [ ] 复位判据为“系统更新时间显示停留超时 **且** 内存活任务表查无此记录”，二者缺一不可——只看墙上时钟会误杀正在跑的慢任务，并从后门重开 §4.4 的陈旧覆盖；不新增 `评分开始时间` 字段。

1. **已确认：** 每次评分任务开始后重新读取记录快照，从现有 `需求文档` 字段解析一个或多个飞书 `wiki`、`docx` 或旧版 `docs` 链接，不新增字段。每个链接形成独立的在线文档材料；任一文档解析或导出失败都会阻断整次采集。
2. **已确认：** 支持类型以飞书 Drive v1 导出任务 API 能否导出 PDF 为准。官方接口当前只有旧版 `doc` 和新版 `docx` 支持 PDF；每个 Wiki 链接必须先解析真实对象，且 `obj_type` 为 `doc/docx` 才受支持。`sheet`、`bitable`、`slides`、`mindnote`、普通文件等不进入评分。参考：[飞书创建导出任务](https://open.feishu.cn/document/server-docs/docs/drive-v1/export_task/create)。
3. **已确认：** 任何可评分提交都会生成总 PDF，因此评分模型必须具备 PDF/视觉文件能力。纯文本模型属于能力不匹配，必须在调用 AI 前终止。
4. **已确认：** 文件能力模型导出 PDF 失败时视为内容采集不完整；本次评分失败并进入重试，不允许调用 `raw_content` 纯文本接口或使用旧缓存补位。
5. **已确认：** 所有实时内容采集成功后生成不可变的 `CollectedContent`，评分过程只使用该次任务的快照。
6. **已确认：** 在线文档实时导出失败时，不得使用已有 `文档内容缓存` 作为评分输入。
7. **已确认：** `文档内容缓存` 保存本次任务中“在线文档 + 全部附件”的合并转写，不包含 `原始描述`，也不作为评分输入或读取失败时的兜底。每次完整采集并成功生成合并转写后，都覆盖现有缓存，不再采用“缓存非空则跳过更新”的逻辑，也不新增 `文档缓存版本` 字段。合并转写属于评分成功的必要步骤；转写失败时不得写回评分或增加修改轮次，整次任务按技术失败重试。
8. **已确认：** 用户点击“重新评分”后必须重新执行以上流程，不能复用上一次任务的文档内容。
9. **已确认：** 原始描述、在线文档和所有附件放入同一个 `ScoringPayload`，通过一次多模态模型调用生成一个综合分数，不分别评分或计算平均值。
10. **已确认：** 在线文档只以导出的 PDF 参与评分；`raw_content` 不用于评分、缓存或任何降级路径。
11. **已确认执行顺序：** `完整采集 → 合并转写成功 → AI 综合评分成功 → 单次写回分数、状态、轮次和文档内容缓存`。任一步失败都不执行后续步骤，也不留下本次部分最终结果。
12. **已确认：** `.doc/.docx` 附件必须成功转换为 PDF 后才能进入评分和合并转写。转换失败时整次内容采集失败并重试，禁止抽取纯文本后降级评分。
13. **已确认并验证：** 当前部署依赖中的 LibreOffice 能将 `.doc/.docx/.txt/.md` 以及 PNG、JPEG、WebP 图片转换为 PDF，无需新增 Python 图像或 PDF 生成依赖。所有支持的附件统一处理为：已有 PDF 直接使用，其余支持格式先转换为 PDF；转换失败时整次内容采集失败，不允许以纯文本或图片模态降级评分。现有 `docx_convert.py` 需要泛化为通用附件转 PDF 组件。
14. **已确认：** 所有在线文档和附件形成 PDF 后，按固定顺序合并为一个总 PDF。合并成功后的同一份总 PDF 同时用于合并转写和综合评分，不向模型发送多份独立文件；合并失败时整次内容采集失败。当前项目没有 PDF 合并依赖，实施时新增 `pypdf`。
15. **已确认：** 合并时在每份材料正文前插入一页来源分隔页，标明材料类型以及在线文档标题或附件原文件名。分隔页属于总 PDF 的一部分，评分模型和合并转写都能据此识别来源边界。

### 本轮评审新确认的决策

- 不新增 `评分开始时间`、`评分触发来源`、`文档缓存版本` 三个多维表字段；分别使用系统更新时间、任务内上下文和“实时失败禁用旧缓存”替代。
- 文件能力模型未能成功导出在线文档 PDF 时，整次内容采集失败；不调用 `raw_content` 或旧缓存降级。
- 所有可评分提交都会形成总 PDF，因此只能由具备 PDF/视觉文件能力的模型评分。
- 在线文档支持范围以飞书官方 PDF 导出能力为准；当前只支持 `doc/docx` 及最终指向它们的 Wiki 节点。
- 原始描述、在线文档和全部附件采用一次模型调用进行综合评分，不拆分单项评分，也不计算平均分。
- 在线文档处理完全删除 `raw_content` 调用；总 PDF 是评分和缓存转写的唯一文档内容来源。
- `文档内容缓存` 保存在线文档与全部附件的合并转写，不包含 `原始描述`，且永不作为评分兜底。
- 每次成功生成合并转写后都覆盖 `文档内容缓存`，不因已有缓存非空而跳过更新。
- 合并转写是评分成功的必要步骤；转写失败时本次评分结果作废，不写回分数、不增加轮次，并按技术失败重试。
- 评分执行顺序固定为“完整采集 → 合并转写 → 综合评分 → 单次最终写回”；前置失败不得产生部分最终结果。
- Word 附件必须成功转换为 PDF；转换失败时整次采集失败，不允许抽取纯文本降级评分。
- `.txt/.md` 与 Word 附件统一转换为 PDF 后参与评分和合并转写；任一转换失败均使整次采集失败。
- PNG、JPEG、WebP 图片附件也统一转换为 PDF，不再以独立图片模态发送；转换失败时整次采集失败。
- 在线文档与全部附件 PDF 合并成一个总 PDF；评分与合并转写必须使用完全相同的总 PDF。
- 总 PDF 在每份材料前插入来源分隔页，标明在线文档标题或附件原文件名，以保留材料边界。
- 总 PDF 顺序固定为在线文档按字段出现顺序在前，附件按多维表字段返回顺序在后。
- 只有 `原始描述` 时不允许评分；需提示用户上传在线需求文档或附件，且不增加修改轮次。
- 一个或多个在线需求文档与附件至少存在一项即可评分，不要求文档和附件同时存在。
- `原始描述` 作为总 PDF 外的独立文本提示参与评分，不进入总 PDF和文档内容缓存。
- 支持与不支持附件混合存在时整次不评分；一次列出全部不支持文件，用户修复后手动重评。
- 损坏、加密或无法解析的附件属于用户可修复问题；整次不评分，列出问题文件后等待替换和手动重评。
- 在线需求文档无权限、失效、类型不支持或已删除属于用户可修复问题，进入“未通过”而非“评分异常”。
- 删除 notification.py 业务频控（60min 冷却 / 每日 3 条）；轮次改由多维表字段承载。
- 轮次 `+1` 只在终态发生、中途异常不计数；先算轮次再选状态，达上限即时写“已驳回”并发无按钮卡片。
- 崩溃恢复由后台清道夫承担：周期扫描 + “超时且内存无活任务”判据 + 复位后直接入队。
- 采用点击即等待模型（走位 C），删除“评分期间内容变化”守卫与全部内容签名；改用 fencing 防清道夫诱发的僵尸写回。
- 管理员重试与用户重评走同一协调器入口 / 同一记录锁 / 同一幂等；卡片置灰仅为 UX。
- 瞬时技术故障自动重试 3 次；全部失败后写为“评分异常”，只通知管理员群，不通知提交人且不占轮次。
- 技术重试只重试失败的单步操作，复用本任务中已经成功产生的中间结果，不从头重跑整个流程。
- `需求文档` 字段可解析多个在线文档，任一文档不可用都会阻断整次评分。
- 在线文档按解析后的真实 token 去重；Wiki 与直链指向同一文档时只保留先出现项，且不得改变其他文档的顺序。
- 附件按 `file_token` 去重，只保留第一次出现位置且不得改变其他附件的顺序。

### 风险接受栏（明确认下、不再防护）

- **不做用户级限流**：理由是内部小组提报量小，不会出现大量并发提报；仅保留发送侧熔断防回归循环打爆整个 bot。
- **管理员授权 = 群成员身份，不做 open_id 名单校验**：依赖内部小组可信。将来该群拉入外部人员时，这道门是敞开的，需届时补名单校验。

剩余决策确认前，可以先完成测试基线、卡片回调骨架、AI Schema 校验、同步 SDK 隔离和状态写回检查。
